1. ARCHITECTURE LAYERS — Describe the role of each layer with specificity: (1) Albert AI Layer — what it requests and what it never does directly; (2) API Gateway — authentication, rate limiting, routing, logging responsibilities; (3) Data Service Layer — caching strategy, translation from modern requests to legacy formats, abstraction contract; (4) Legacy Financial Systems — Bursar OLTP, Financial Aid System, Registrar DB, General Ledger — and why none of these change their interfaces for Albert:
(1) Albert AI Layer
The Albert AI Layer is the user-facing explanation and guidance layer. It interprets structured responses, explains holds, summarizes eligibility results, guides payment or aid workflows, and routes users to the right office or next step. It may prepare case summaries or explain what happened, but it never directly queries the Bursar, Financial Aid, Registrar, or the General Ledger. It also never changes official records, clears holds, posts payments, or makes binding financial or academic decisions on its own. If evidence is incomplete or confidence falls below the required threshold, it must escalate rather than guess. (2) API Gateway
The API Gateway is the controlled front door between Albert and the integration services. It validates authentication and session tokens, enforces role-based access control, rate-limits high-volume calls, routes requests to the correct service, assigns correlation IDs, and writes audit logs for every request. It should also mask sensitive values in logs and reject any non-approved endpoint. The gateway does not apply business rules and does not store authoritative financial data. (3) Data Service Layer
The Data Service Layer is the orchestration and abstraction layer. It translates Albert’s modern JSON/REST requests into whatever the legacy systems can support, aggregates multi-system responses, enforces freshness rules, manages short-lived caches, and returns one normalized response contract to Albert. This is where registration eligibility is assembled from several systems at once. It also owns graceful degradation logic, partial-response handling, stale-data labeling, and adapter logic so Albert is not coupled to legacy schemas. It may cache data such as balance summaries or degree-audit views, but it never modifies any legacy database directly. (4) Legacy Financial Systems
The legacy systems remain the systems of record.
Bursar OLTP owns charges, balances, payments, refunds, and financial holds.
The Financial Aid System owns awards, disbursements, and pending aid items.
Registrar DB owns academic standing, enrollment status, registration eligibility, and related restrictions.
The general ledger / ERP owns enterprise financial posting and must never be directly touched by Albert.

4. DESIGN RATIONALE — Why did you choose this layered architecture over simpler alternatives (e.g., direct database queries, single middleware)? Address: performance, security, and maintainability trade-offs:
I chose this layered architecture because it gives the best balance of performance, security, and maintainability.

From a performance perspective, direct AI-to-legacy access would overload OLTP systems that were never designed for conversational or high-frequency portal traffic. The Data Service Layer lets NYU use short caches, parallel orchestration, and source-specific freshness rules so high-value actions stay responsive without hammering the Bursar or Registrar. The Module 10 guidance explicitly separates OLTP, OLAP, and AI layers for this reason.

From a security and governance perspective, the gateway centralizes authentication, RBAC, audit logging, and rate limiting, while the AI remains read-and-assist only. That matches your prior guardrails: the bot does not change records, does not make final decisions, and must escalate when confidence or evidence is insufficient. This is much safer than direct database queries or a loosely controlled single middleware layer.

From a maintainability perspective, the Data Service Layer isolates Albert from the schema quirks and lifecycle of PeopleSoft/Oracle-style legacy systems. If a legacy interface changes, NYU updates the adapter, not the AI layer, UI, and every downstream feature. That also makes auditing and procurement review much easier because the contract boundary is clean: Albert consumes approved services, and official systems remain authoritative.

2. REGISTRATION ELIGIBILITY ENDPOINT — Your most complex call. Describe in full: what sub-calls it orchestrates simultaneously, how it handles partial responses (e.g., holds loaded but academic standing unavailable), how long it may take, and what timeout threshold triggers a fallback:
GET /students/{id}/registration-eligibility is the most complex endpoint because it orchestrates several sub-calls at once. The required calls are:

GET /students/{id}/holds
GET /students/{id}/academic-standing
GET /students/{id}/enrollment-status

The optional explanatory calls are:

GET /students/{id}/degree-audit-summary
GET /students/{id}/financial-aid-summary

The Data Service Layer should execute the required calls in parallel, then apply rule logic in this order:

If a blocking financial hold exists → return blocked_financial.
Else if standing or enrollment restrictions block registration → return blocked_academic_or_enrollment.
Else if any required source is missing → return review_required.
Else, return eligible.

Partial-response handling must be strict. If holds are loaded but academic standing is unavailable, Albert must not approve registration. It should return a review_required result because a multi-factor decision cannot be safely made from incomplete required data. If only financial aid is stale or missing, the endpoint may still return an eligibility result, but the aid portion must be labeled informational only. This follows both your prior Group 1 failure logic and your AI guardrail rule that missing evidence should trigger escalation, not improvisation.

For timing, I would set sub-call timeout = 700 ms and overall orchestration timeout = 1.5 seconds, with a hard fallback at 2.0 seconds. That keeps the endpoint realistic for a more complex multi-system call while still staying close to the course guidance that live portal financial lookups should be fast. If the timeout threshold is hit, the endpoint returns a structured fallback response rather than a blank failure.

| Data type | Freshness strategy | Justification |
| ----------------------------------------------------- | ------------------------- | ----------------------------------------------------------------------------------------------------------- |
| Active financial holds + owner office | Real-time | A stale hold can wrongly block or wrongly allow registration. |
| Academic standing + enrollment restriction flags | Real-time | These directly determine whether the student is eligible right now. |
| Registration eligibility decision | Real-time | This is a live gatekeeping decision and cannot rely on stale required inputs. |
| Payment intent creation + payment confirmation status | Real-time | Payment is transactional; stale status can mislead students about whether a hold should clear. |
| Account balance summary | 5-min cache | Important and student-facing, but a short cache protects Bursar OLTP while remaining operationally current. |
| Course seat availability / section status | 5–10 min cache | Students and advisors need a near-current view, but continuous source hits would overload systems. |
| Degree-audit summary / enrollment snapshot | 15-min cache | Important for advising and planning, but usually not second-by-second. |
| Payment history / refund visibility | 15-min cache | Useful for explanation, but not usually a live blocking decision. |
| Billing statements | On-demand + session cache | Rarely accessed compared with holds or balance checks. |
| Financial aid awards / disbursement details | Nightly batch | Changes less frequently and is normally used for planning/explanation, not same-second transaction control. |
| Risk scores / trend analytics / planning dashboards | Nightly batch | These support prioritization and planning, not live transaction execution. |
| Academic calendar / deadlines | Weekly scheduled sync | These change on planned cycles and do not need constant refresh. |
2. REAL-TIME JUSTIFICATION — For each data type you assigned real-time access: what specific student action depends on it being current, and what would happen if data were even 10 minutes stale?
Active holds and hold owners must be real-time because the student may be trying to register or pay right now. If this data is even 10 minutes stale, Albert could say a hold still exists after it was already cleared, or worse, say the student is clear when the hold still blocks registration.

Academic standing and enrollment restriction flags must be real-time because they are authoritative eligibility conditions. If they are 10 minutes stale, Albert may send the student down the wrong path, such as telling them the issue is financial when the real blocker is academic or enrollment-based.

Registration eligibility itself must be real-time because it is the final live answer used at the moment of action. If it is 10 minutes stale, Albert could either wrongly stop a legitimate registration attempt or wrongly allow a non-compliant one.

Payment intent and payment confirmation must be real-time because the student’s next step depends on knowing whether the payment session is active and whether payment actually posted. If this is 10 minutes stale, the student could believe a hold should already be gone when it has not yet been confirmed in the system of record.
3. STALENESS DISCLOSURE DESIGN — For each cached or batch data type, write the exact interface message Albert shows when data may be stale (e.g., 'Balance as of [timestamp]'). How can a student request a manual refresh?
For 5-minute cached balance data:
“Account balance as of [timestamp]. Recent payments, charges, or hold releases from the last 5 minutes may not yet appear.”
Action: Refresh now

For 5–10 minute cached seat availability:
“Seat availability last refreshed at [timestamp]. A section may fill or reopen before the next refresh.”
Action: Recheck seat status

For 15-minute cached degree audit / enrollment snapshot:
“Degree progress and enrollment summary last refreshed at [timestamp]. Recent approvals, exceptions, or record corrections may not yet be reflected.”
Action: Request fresh check

For nightly-batch financial aid data:
“Financial aid information last updated at [timestamp]. Recent award or disbursement changes may not yet be reflected. Please verify details with Financial Aid before making payment or enrollment decisions based on this information.”
Action: Refresh aid status or Contact Financial Aid

For on-demand statements:
“Statement pulled at [timestamp] for this session. New postings after this time may not appear until you refresh.”
Action: Reload statement

A manual refresh should invalidate the session cache for that widget and re-issue the source call through the gateway. If the source is rate-limited or unavailable, Albert should tell the user that the refresh failed rather than silently reuse stale data.
PART 4 · 30 min
Failure Design & Graceful Degradation
1. FAILURE SCENARIO 1 — Bursar API timeout at 11:50pm during peak registration. Describe in full: (a) how Albert detects the failure, (b) exact message shown to the student, (c) what data Albert can still show from cache, (d) fallback path provided (direct contact info, alternative action), and (e) how the system retries:
(a) How Albert detects the failure
The API Gateway detects repeated Bursar sub-call timeout events and returns a 504/error state to the Data Service Layer. The Data Service records the failed dependency, correlation ID, timestamp, and whether cached Bursar data is available.

(b) Exact message shown to the student
“We can’t confirm your Bursar hold status right now because the financial system did not respond. Your last available Bursar snapshot is from [timestamp], but we cannot use cached hold data to approve registration. Please try again now. If this is urgent, use the support options below and keep reference ID [correlation_id].”

(c) What data Albert can still show from cache
Albert may still show the last known balance, last known hold snapshot, and timestamped course basket, but all cached bursar information must be clearly labeled "informational only" and not authoritative for clearance.

(d) Fallback path provided
Albert should provide the following:

Retry now
Save attempted registration basket
Open Bursar/Registrar support contact card
Create urgent review case if NYU enables that workflow

(e) How the system retries
The service should do one immediate silent retry after 300 ms, one retry after 1 second, and then stop and return the graceful degradation screen. A manual retry button can trigger a new attempt, but Albert must not spin indefinitely or silently fail.

What Albert does NOT do
It does not assume “no hold,” does not auto-approve registration from stale cache, and does not leave the student on a blank or endlessly loading screen. That aligns directly with the Module 10 failure case and your Group 1 degradation rule.
2. FAILURE SCENARIO 2 — Financial Aid nightly batch fails, data is now 48 hours stale. Describe: (a) how Albert detects or is notified of the failure, (b) what the student sees and when, (c) what advisors see and when, (d) how you prevent a student from making a financial decision based on stale aid data:
(a) How Albert detects or is notified
The nightly ETL/batch job writes a freshness heartbeat after each successful sync. If the heartbeat exceeds 24 hours, the aid dataset gets a stale flag; if it exceeds 48 hours, the stale flag becomes critical.

(b) What the student sees and when
At 24 hours stale, the student sees a warning banner. At 48 hours stale, Albert shows a stronger banner:
“Financial aid information may be out of date. This data was last refreshed 48 hours ago due to a system update delay. Your current award or disbursement status may have changed. Please verify details with Financial Aid before making payment or enrollment decisions based on this information.”

(c) What advisors see and when
Advisors see the same stale-data banner plus a case-level flag on affected students and a dashboard indicator showing that aid-dependent advice should be treated cautiously.

(d) How you prevent a student from making a financial decision based on stale aid data
Albert must mark the aid panel informational only, avoid words like “confirmed” or “cleared,” and suppress any AI conclusion that depends on fresh aid data. For example, Albert should not imply that aid will definitely clear a balance or hold until fresh data is available. It should route the student to financial aid or the bursar for confirmation. This follows your earlier Group 1 answer almost exactly.
3. FAILURE SCENARIO 3 — Registration eligibility check returns partial data: holds loaded from Bursar but academic standing unavailable from the Registrar. Justify your design choice: does Albert (a) approve, (b) block with explanation, or (c) escalate to human? Write the exact message the student sees and the exact escalation path:
Design choice: (c) escalate to human / review required

Albert should not approve because registration eligibility is a multi-factor decision and one required source is missing. It should also not hard block as ineligible because the missing data does not prove the student is disqualified. The safest choice is a review-required outcome.

Exact message shown to the student
“Albert could not fully verify your registration eligibility because academic standing information is temporarily unavailable. We found your hold status, but we do not have all required data to approve registration. Please try again shortly or contact the Registrar if the issue is urgent.”

Exact escalation path

Return review_required from GET /students/{id}/registration-eligibility
Create an incident/case with correlation ID and partial source results
Route the case to the Registrar review queue
If the student has an assigned advisor, notify that advisor too
Offer the student a "Retry in 5 minutes" button.

This is the same logic your Group 1 finance design used: missing required data should trigger safe escalation, not approval by assumption.
4. GRACEFUL DEGRADATION FRAMEWORK — State your overarching graceful degradation principle as a formal design rule. Then list the 5 things your design ALWAYS does in any failure state, regardless of which system failed:
Formal design rule:
When required source data is unavailable, stale, or incomplete, Albert must fail safely by being transparent about the limitation, avoiding authoritative decisions, and directing the user to the next best verified path.

The 5 things Albert ALWAYS does in any failure state:

States exactly what failed — name the missing or stale source, not just “something went wrong.”
Shows what is still safe to show — partial data or cached data, always with timestamps.
Avoids guessing—no authoritative decision if required inputs are missing.
Provides a next step — retry, support contact, human review, or alternative verified path.
Logs and monitors the incident—correlation ID, audit trail, alerts, and retry behavior.
